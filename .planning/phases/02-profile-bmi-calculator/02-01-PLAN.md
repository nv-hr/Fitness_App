```sql
CREATE TABLE IF NOT EXISTS profiles (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL UNIQUE,
  weight_kg DECIMAL(5,2) NOT NULL,
  height_cm DECIMAL(5,2) NOT NULL,
  age INT NOT NULL,
  gender ENUM('male', 'female', 'other') NOT NULL,
  fitness_goal ENUM('lose_weight', 'maintain', 'gain_weight') NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB;
```

1. Create backend/src/services/profile.service.js with four functions:

- `calculateBmi(weightKg, heightCm)` — Pure function. Per D-13: heightM = heightCm / 100, bmi = weightKg / (heightM * heightM). Return bmi rounded to 1 decimal place.

- `getBmiCategory(bmi)` — Per D-14 Asian-Pacific cutoffs: return 'underweight' if bmi < 18.5, 'normal' if bmi >= 18.5 && bmi <= 22.9, 'overweight' if bmi >= 23 && bmi <= 24.9, 'obese' if bmi >= 25.

- `createProfile(userId, profileData)` — profileData = { weightKg, heightCm, age, gender, fitnessGoal }. Validate: weightKg must be 2-300, heightCm must be 50-250, age must be 5-120, gender must be 'male'|'female'|'other', fitnessGoal must be 'lose_weight'|'maintain'|'gain_weight'. Throw ValidationError for invalid values. Call profileRepository.create(). Compute BMI via calculateBmi(). Compute category via getBmiCategory(). Return { profile: createdProfile, bmi, bmiCategory }.

- `getProfile(userId)` — Call profileRepository.findByUserId(). If null, return null. Otherwise compute BMI and category, return { profile, bmi, bmiCategory }.

- `updateProfile(userId, profileData)` — Same validation as createProfile. Call profileRepository.updateByUserId(). Then call getProfile(userId) to return updated data with BMI.

Import profile repository functions as: `import { create as createProfileRepo, findByUserId, updateByUserId } from '../repositories/profile.repository.js'`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Profile data (weight, height, age) submitted as untrusted input |
| JWT→profile routes | authenticateToken middleware validates identity before profile access |
| API→MySQL | Parameterized queries prevent SQL injection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | profile.controller.js | mitigate | Validate all input fields with explicit ranges (weight 2-300, height 50-250, age 5-120) before passing to service layer |
| T-02-02 | Information Disclosure | profile.routes.js | mitigate | authenticateToken middleware on all profile routes — users can only access their own profile via req.user.userId |
| T-02-03 | Repudiation | profile.repository.js | accept | Profile CRUD operations logged via morgan; no audit trail required for v1 health app |
| T-02-04 | Denial of Service | /api/profile routes | mitigate | Rate limiter set to 15 requests per 15 minutes on profile endpoints |
| T-02-05 | Elevation of Privilege | profile.routes.js | mitigate | user_id enforced from JWT (req.user.userId), never from request body — prevents users from modifying other profiles |
</threat_model>
