<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Backend API | Untrusted input: registration data, login credentials, environment variables |
| Backend → MySQL Database | SQL injection risk through user-provided email/password fields |
| Docker Compose → Host | Environment variable exposure via .env file |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | POST /api/auth/register | mitigate | Validate email format with express-validator, enforce password minimum length (8+ chars), reject duplicate emails via UNIQUE constraint |
| T-01-02 | Tampering | user.repository.js SQL queries | mitigate | All queries use parameterized placeholders (?) — no string concatenation. mysql2 prepared statements prevent SQL injection |
| T-01-03 | Information Disclosure | .env file with secrets | mitigate | .env excluded from git via .gitignore. .env.example provides template only. Docker Compose reads from host env vars, not hardcoded values |
| T-01-04 | Denial of Service | MySQL connection pool | mitigate | connectionLimit: 10 prevents connection exhaustion. queueLimit: 0 rejects excess connections immediately rather than queuing indefinitely |
| T-01-05 | Elevation of Privilege | Database user permissions | mitigate | MySQL user (DB_USER) has only CRUD permissions on fitness_app database. Root password (DB_ROOT_PASSWORD) separate and not used by application |
</threat_model>
