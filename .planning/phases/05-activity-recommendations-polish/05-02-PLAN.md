Export these functions:

**getRandomActivities(userId, goalTags, count = 5)**:
- Query activities where JSON_CONTAINS(goal_tags, tag) for ANY tag in goalTags array
- Use ORDER BY RAND() for shuffle (per D-45: simple randomization, no state)
- LIMIT to count (default 5)
- Return array of activity objects
- SQL pattern: `SELECT * FROM activities WHERE JSON_OVERLAPS(goal_tags, JSON_ARRAY(...)) ORDER BY RAND() LIMIT ?`
- MySQL 8.4 supports JSON_OVERLAPS for array intersection

**getAllActivities(userId, goalTags)**:
- Query all activities filtered by goal_tags (for the activity pool browser on /activities page)
- ORDER BY name ASC
- Return full array

**getActivityById(activityId)**:
- SELECT * FROM activities WHERE id = ? LIMIT 1
- Return single activity or null

All functions are user-scoped where relevant (userId parameter for future activity logging), but the activities table itself is global (no user_id column).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → /api/activities/* | Untrusted HTTP requests; authentication via JWT cookie required |
| Controller → Service | Internal trust, but fitness_goal must come from user's profile (not client input) |
| Repository → MySQL | Parameterized queries prevent SQL injection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Spoofing | activity.routes.js | mitigate | All routes behind authenticateToken middleware — rejects unauthenticated requests with 401 |
| T-05-05 | Tampering | activity.controller.js | mitigate | fitness_goal read from user's profile via repository (not from request body/query) — prevents user from manipulating recommendation filter |
| T-05-06 | Information Disclosure | activity.repository.js | accept | Activities table is global (no user-scoped data); all activities are public seed data — no disclosure risk |
| T-05-07 | Denial of Service | activityLimiter in app.js | mitigate | Rate limiter set to 60 req/15min — prevents abuse of ORDER BY RAND() queries which are computationally expensive |
| T-05-08 | Repudiation | user_activity_log table | accept | Activity logging is optional (D-44); no audit trail required for v1 recommendations |
</threat_model>
